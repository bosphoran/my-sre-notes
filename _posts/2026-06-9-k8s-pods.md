## Introduction

A Pod is the smallest deployable unit representing a single instance of an application. Think of pods as a box that can hold one or more containers together.

- Each pod gets a unique IP address.
- Containers inside a pod communicate using **localhost** but with different port numbers.
- Containers inside a pod share the same volume mount.

![pod]({{ site.baseurl }}/assets/img/k8s-course/pod.png)

We can create a pod in two ways:

- **Imperative Commands**: Allows to create k8s objects from the command-line without needing to write configuration file.
- **Declarative Approach**: You write a **YAML** file that describes the desired state of your application. This file is called a manifest.

### Creating Pod Using Imperative Commands

Here, we will use **kubectl run** command for creating pod using imperative commands.

Lets create an **Nginx** pod with the following details:

- The name of the pod is webserver-pod
- Use nginx container image. Tag should be 1.14.2
- It should have restart policy Never.
- It should have labels app: webserver and environment: production
- Add an annotation named description and describe the pod.

```bash
vagrant@controlplane:~$ kubectl run webserver-pod --image nginx:1.14.2 --restart=Never --labels=app=webserver,environmen
t=production --annotations description="This pod runs the Nginx web-server."

pod/webserver-pod created
```

The above command will directly deploy the pod in the clusters default namespace. Lets verify our pod:

```bash
vagrant@controlplane:~$ kubectl get pods

webserver-pod   1/1     Running   0          62s
```

### Running Interactive Pod For Testing

An **interactive pod** allows you to spin up a temporary container with terminal access to debug, test network connectivity, or explore cluster environments.

While running interactive pod, use the **--rm** flag to ensure the pod is deleted after the command is finished.

```bash
vagrant@controlplane:~$ kubectl run -i --tty testpod --image=busybox --rm --restart=Never -- sh
All commands and output from this session will be recorded in container logs, including credentials and sensitive information passed through the command prompt.
If you don't see a command prompt, try pressing enter.
/ # ls
bin    dev    etc    home   lib    lib64  proc   root   sys    tmp    usr    var
/ # hostname
testpod
/ # cd var/
/var # ls
run    spool  www
/var # exit
pod "testpod" deleted from default namespace
```

> You may run an interactive pod without **--tty** flag. However, **tty** which stands for Teletypewriter, provides a proper terminal emulation.

### Creating Pod Using Declarative Approach

For **declarative approach**, first we create a **yaml file**, modify it and apply it to create our pod.

The **yaml** file will not include all our pod specifications, but it will provide us with the basic pod template that we can modify based on the requirements.

```bash
vagrant@controlplane:~$ kubectl run webserver-pod --image=nginx --dry-run=client -o yaml > pod.yaml
```

The above command creates the **pod.yaml** file in the current directory.

- **--dry-run=client** flag performs a client-side dry run by pretending to deploy an object in the cluster and validate it.
- **-o yaml** indicates that we want the output of the command in the YAML format.

Lets view the generated pod manifest file:

```bash
vagrant@controlplane:~$ cat pod.yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: webserver-pod
  name: webserver-pod
spec:
  containers:
  - image: nginx
    name: webserver-pod
    resources: {}
  dnsPolicy: ClusterFirst
  restartPolicy: Always
status: {}
```

### Deploy the Pod Manifest

Using this manifest file, we can change the name of the pod or add or remove any field that is not relevent or required.

Lets remove the **dnsPolicy**, **restartPolicy** and **status** fields from the yaml file.

```bash
vagrant@controlplane:~$ cat pod.yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: webserver-pod
  name: webserver-pod
spec:
  containers:
  - image: nginx
    name: webserver-pod
    resources: {}
```

Now we can edit this manifest file and create the pod using the **kubectl apply** command.


```bash
vagrant@controlplane:~$ kubectl apply -f pod.yaml
pod/webserver-pod created
vagrant@controlplane:~$ kubectl get pods
NAME            READY   STATUS    RESTARTS   AGE
webserver-pod   1/1     Running   0          21s
```

Lets **delete** the pod.

```bash
vagrant@controlplane:~$ kubectl delete -f pod.yaml
pod "webserver-pod" deleted from default namespace
vagrant@controlplane:~$ kubectl get pods
No resources found in default namespace.
```

### Summary

- To generate the basic Pod template, use the command:
  - **kubectl run webserver-pod --image=nginx --dry-run=client -o yaml > pod.yaml**
  - The **--dry-run=client** flag performs a client-side dry run to validate deployment without actually deploying it.
  - The **-o yaml** flag outputs the result in YAML format.
- Once the YAML file is generated, you can edit the manifest file based on your requirements using the **kubectl apply** command.
- Edit the YAML file by customizing fields, changing existing values, or removing/adding fields as needed.
- After editing the YAML file, apply the manifest using the command:
  - **kubectl apply -f pod.yaml**
- If you want to delete the Pod, use the command:
  - **kubectl delete pod webserver-pod**
  - **kubectl delete -f pod.yaml**

### Multi Container Pods


Multi-container pods share the same lifecycle, network space, and storage volumes. Simply put, they are created and destroyed together.

Let's create a multi-container pod for the following use case.

- A pod with two containers.
  - A **webserver** container using **nginx** image.
  - A **log-reader** container using **busybox** image.
- webserver container **mounts** the log files to pods **nginx-logs** shared volume at **/var/log/nginx**.
- The log reader container mounts to the same shared volume at **/var/log/nginx** so that it has access to the log files written by the webserver container.

```bash
vagrant@controlplane:~$ kubectl run multi-container-pod --image nginx --dry-run=client -o yaml > multi-container-pod.yaml
```

Open the **multi-container-pod.yaml** file using an editor and add another container and volume details.

```bash
vagrant@controlplane:~$ cat multi-container-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: multi-container-pod
  name: multi-container-pod
spec:
  containers:
  - image: nginx
    name: webserver
    resources: {}
    volumeMounts:
    - name: nginx-logs
      mountPath: /var/log/nginx
  - image: busybox
    name: log-reader
    command: ["/bin/sh"] # Override default entrypoint: start a shell
    args: ["-c", "tail -f /var/log/nginx/error.log"] # Pass arguments to the shell:
                              # -c tells /bin/sh to run the following command
                              # "tail -f ..." continuously reads the nginx error log
    volumeMounts:
    - name: nginx-logs
      mountPath: /var/log/nginx
  volumes:
   - name: nginx-logs
     emptyDir: {}
  dnsPolicy: ClusterFirst
  restartPolicy: Always
status: {}
```

- We have added a **log-reader** container using the **busybox** image and added volume mount, custom **command** and **args** that tails the nginx error log file.
- We're also creating an **empty directory** type volume named nginx-logs and mounting it to both the containers.

Now, lets apply the **multi-container-pod.yaml** or manifest file and check the status and logs.

```bash
vagrant@controlplane:~$ kubectl apply -f multi-container-pod.yaml
pod/multi-container-pod created

# Lets verify the pod

vagrant@controlplane:~$ kubectl get pods -o wide
```

### Example Scenario 01

Create a Pod named webserver. The Pod should use the nginx image and the tag should be 1.25.4 . The container name should be webserver-container and use the OnFailure restart policy. Label the pod with type=web-application.

```bash
vagrant@controlplane:~$ kubectl run webserver 
                                    --image=nginx:1.25.4 
                                    --overrides='{"containers": {"name": "webserver-container"}}' --restart=OnFailure --labels="type=web-application"
```

```bash
vagrant@controlplane:~$ kubectl get pods -o wide
NAME                  READY   STATUS    RESTARTS   AGE   IP               NODE     NOMINATED NODE   READINESS GATES
multi-container-pod   2/2     Running   0          26m   10.244.140.83    node02   <none>           <none>
webserver             1/1     Running   0          98s   10.244.196.158   node01   <none>           <none>
```

You may also use the **--dry-run=client** and **-o yaml** flags to generate the basic YAML template for your Pod. Then, edit the yaml file and apply it to create you pod.

```bash
kubectl run webserver 
            --image=nginx:1.25.4 
            --overrides='{"containers": {"name": "webserver-container"}}' 
            --restart=OnFailure 
            --labels="type=web-application" 
            --dry-run=client -o yaml > webserver.yaml
```

### Example Scenario 02

Create a **multi-container** pod named **busybox-multi-container**. There will be **3** busybox containers. All containers should run the below commands:

- First Container (container-1): “**ls; sleep 3600;**”
- Second Container (container-2): “**echo This is techiescamp; sleep 1000;**”
- Third Container (container-3): “**date; sleep 3600;**"
- Once in the running state, check the logs of each container and forward the logs to three different files:
  - container1.log, container2.log, container3.log

```bash
vagrant@controlplane:~$ kubectl run busybox-multi-container --image=busybox --dry-run=client -o yaml > busybox-multi-container.yaml
```

Now, open the **busybox-multi-container.yaml** and add the required containers and commands.

```bash
vagrant@controlplane:~$ cat busybox-multi-container.yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: busybox-multi-container
  name: busybox-multi-container
spec:
  containers:
  - image: busybox
    name: container-1
    resources: {}
    command: ["/bin/sh"]
    args: ["-c", "ls; sleep 3600;"]
  - image: busybox
    name: container-2
    resources: {}
    command: ["/bin/sh"]
    args: ["-c", "echo This is zacademy.org; sleep 1000;"]
  - image: busybox
    name: container-3
    resources: {}
    command: ["/bin/sh"]
    args: ["-c", "date; sleep 3600;"]
  dnsPolicy: ClusterFirst
  restartPolicy: Never
status: {}
```

Now, lets apply our yaml file to create the pod.

```bash
vagrant@controlplane:~$ kubectl apply -f busybox-multi-container.yaml
pod/busybox-multi-container created
```

Verify pod status:

```bash
vagrant@controlplane:~$ kubectl get pods
NAME                      READY   STATUS    RESTARTS   AGE
busybox-multi-container   3/3     Running   0          56s
multi-container-pod       2/2     Running   0          73m
webserver                 1/1     Running   0          48m
```

Finally, lets forward the logs of each container to a different file.

```bash
# Get container-1 logs
```

As you can see, each container logs the command specified in the **args** field of the manifest file.
